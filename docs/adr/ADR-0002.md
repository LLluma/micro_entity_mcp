# ADR-0002: Durable attributed-entity store (decision-log / ADR instance)

## Status

Accepted (2026-06-30).

**Resolution note (2026-06-30):** the open question on engine sharing (see
Neutral consequences) is resolved by **ADR-0003** — the generic engine is a
shared internal monorepo module; this instance is a thin profile server over
it. This record otherwise remains in force; it is *refined*, not superseded.

**Refinement note (2026-06-30):** the "structural fields (`status`, timestamps)"
wording is refined by **ADR-0004** — `status` is a designated attribute governed
by the registered value-set, not a structural field. Core structural fields are
now `id` + timestamps only. Still refined, not superseded.

**Schema note (2026-06-30):** the concrete record schema and status lifecycle left
open here are settled by **ADR-0005** — the rich-frontmatter on-disk shape is
canonical, status enum is `{Proposed, Accepted, Superseded}`, and supersession is a
`superseded_by` attribute pointer. Refined, not superseded.

## Context

A durable, append-only, searchable store of architectural decisions is needed —
a private decision log, not a team artifact, because client repositories do not
accept it. Forces shaping the decision:

- **Durable, append-only, searchable.** The opposite lifecycle to the ephemeral
  store (ADR-0001): records are never deleted, outlive a session and a project,
  and the primary operation is search across time.
- **Cross-cutting by nature.** A decision that changes interaction between
  pipeline components belongs to no single repository — it sits above them.
  This rules out per-repo storage.
- **Dumb window.** The tool must expose data and validate form, but never emit
  verdicts (coverage, adequacy, completeness). All judgment stays in the
  caller, in the open, with no oracle in disguise.
- **Data security.** Client policy forbids third-party models for data
  processing; any tool that ships record content to an external LLM is
  disqualified.
- **Non-intersection (key architectural premise).** This store knows nothing
  about the ephemeral store. Relationships between their entities live *outside*
  both: an entity carries generic attributes that merely *expose* a link to an
  external reader/agent; the store never resolves or interprets them.

Existing solutions were reviewed; none is a clean fit (see Alternatives).

## Decision

Use the **same generic attributed-entity MCP engine** as ADR-0001, deployed
here as the durable (decision-log) instance. The engine has no domain
knowledge; "ADR" is a deployment profile.

**Entity model.** An entity = an id + server-understood structural fields
(`status` enum: Proposed / Accepted / Superseded / …; timestamps) + a bag of
**generic attributes** (project, topic tags, supersedes pointer, relates_to,
etc.). The server validates *form* only and assigns **no meaning** to attribute
keys or values.

**Tools — access and state, zero interpretation:**
- `add`, `get`, `list`, `update` (status).
- `query`: the same primitive membership filter as ADR-0001 (within-attribute
  OR, across-attribute AND; NOT / cross-attribute OR / diff composed by the
  caller). It returns full records (a **wide dumb window**) so any slice the
  caller needs is obvious without per-item perception.
- `search`: full-text at first; may graduate to vector search on the existing
  `mc_rag_doc` stack (Docling + LanceDB + bge-m3) — the "hard" part is already
  written and understood.

**Append-only.** New records supersede old ones (a generic `supersedes`
attribute); records are never edited or deleted.

**Optional generic integrity.** A per-attribute registered value-set may be
enforced for a chosen attribute key — generic form validation, applied to
whatever attribute the caller uses for cross-cutting membership, to prevent
typo-drift in queries. The server does not know that attribute *means*
"project"; it only enforces a registered set.

**Multi-membership.** Any attribute may hold multiple values, so a record may
belong to several cross-cutting groups at once. The store is centralized and
out of git; partitioning is by attribute, not by repo.

**Relationships are external.** Coverage ("do the ephemeral entities implement
this decision?"), dangling references, and similar are caller-side joins across
two independent stores. This store has no knowledge of the other; a relationship
is exposed only through generic attributes that an external reader interprets.
Semantic judgment (coverage adequacy, "is this the right decision") stays in
the model; the store only provides cheap, wide, dumb access so judgment runs on
retrieved facts, not on perception.

## Consequences

### Positive
- Fits our model; dumb access means semantics are single-owned by the caller —
  no competing definitions of "coverage" drifting apart.
- Cross-cutting + out-of-git are satisfied natively, not by bending a per-repo
  model.
- Data security: zero external calls; all traffic local.
- Survives methodology churn (the orchestration skill has gone through ~15
  revisions): raw data access is orthogonal to *how* we currently think about
  coverage.

### Negative / Costs
- We maintain it.
- Search may need to graduate from full-text to vector (separate work, though
  on an existing stack).

### Neutral
- This very record will be among the first the store holds (dogfooding). The
  `relates_to` attribute on these ADRs is itself a generic external-link
  attribute — the store does not interpret it; an external reader follows it.
- ~~Shares the engine with ADR-0001; the two differ only by deployment
  profile.~~ **Resolved by ADR-0003:** shared `micro_entity` core in a
  monorepo; this instance is the thin profile server `servers/adr.py`, deployed
  as its own idle stdio process.

## Alternatives Considered

### tosin2013/mcp-adr-analysis-server — rejected
73 tools, AI-driven analysis with confidence scoring, tree-sitter AST. Two
disqualifiers: (1) it **requires an OpenRouter API key** — ships content to a
third-party LLM, breaking client data policy; (2) an interpreter-with-verdicts —
the antithesis of a dumb window.

### joshrotenberg/adrs — rejected (closest clean fit)
Rust CLI + MCP (15 tools), adr-tools compatible, tags, full-text search, single
binary; genuine data access in our spirit. Rejected: **repo-scoped** (ADRs as
files in `docs/adr`, project `cwd`) — fights cross-cutting + out-of-git.
Possible fallback if pointed at a central directory and it tolerates
cross-project use; verify before committing to DIY.

### Archyl (SaaS) — rejected
56 tools, C4 + ADR, but hosted and phones home — outside data policy.

### ADR-template agent skills — rejected
Prose conventions for *writing* ADRs, not a data layer; does not address access.
