# ADR-0001: Ephemeral attributed-entity store (TODO instance)

## Status

Accepted (2026-06-30).

**Resolution note (2026-06-30):** the open question on engine sharing (see
Neutral consequences) is resolved by **ADR-0003** — the generic engine is a
shared internal monorepo module; this instance is a thin profile server over
it. This record otherwise remains in force; it is *refined*, not superseded.

**Refinement note (2026-06-30):** the "structural fields (`status`, ordering,
timestamps)" wording is refined by **ADR-0003** (ordering → todo server) and
**ADR-0004** (`status` is a designated attribute governed by the registered
value-set, not a structural field). Core structural fields are now `id` +
timestamps only. Still refined, not superseded.

## Context

The orchestrator loop needs an ephemeral process-state store: what step is
next, mark a step done, is the plan complete. Forces shaping the decision:

- **Out of git.** Client repositories do not accept TODO/ADR artifacts. They
  are currently deleted + squashed before push — manual work and a leak risk.
- **Weak local orchestrator.** Empirically, local models fail on *judgment*,
  not on *bookkeeping* (e.g. recency-bias: "read all items, started from the
  last" — latitude where a procedure should rule).
- **Nature of the entities.** Ephemeral, per-session/per-process, flat, no
  search, discardable.
- **Non-intersection (key architectural premise).** This store and the durable
  store (ADR-0002) **do not know about each other**. Any relationship between
  their entities lives *outside* both stores: an entity may carry a generic
  attribute whose value happens to be an id from another store, but this store
  assigns that attribute no meaning and never resolves it. Relationships
  (reference, coverage, dangling) are composed by the caller via membership
  queries against both stores.
- **Near-zero development cost.** A simple, fully understood spec; codegen
  yields exactly the desired shape with no adaptation rent to a foreign model.

Adoption of Beads (an agent-oriented issue tracker) was considered; it
independently arrives at the same thesis (structured queryable store over
markdown).

## Decision

Build a **generic attributed-entity MCP**, deployed here as the ephemeral
(TODO) instance. The engine has no domain knowledge; "TODO" is a deployment
profile, not a baked-in concept.

**Entity model.** An entity = an id + a small set of server-understood
structural fields (`status`, ordering/sequence, timestamps) + a bag of
**generic attributes** (`key -> scalar | list`). The server validates *form*
only (id well-formed, status in enum, attributes well-typed); it assigns **no
meaning** to attribute keys or values.

**Tools — access and state, zero interpretation:**
- `create`, `get`, `list`, `update` (status), `delete` / `clear`.
- `next`: deterministic — first not-done entity by order (`sorted[0]`), never
  "judge the list".
- `is_complete`: structural — all entities done.
- `query`: a **primitive membership filter** over arbitrary attributes —
  `query({attr: [values], ...})` returns full entities whose attribute `attr`
  intersects `[values]`. Within an attribute: OR (set membership). Across
  attributes: AND (conjunction). Everything else — NOT, cross-attribute OR,
  diff, nesting — is composed by the caller. No expression tree.

**Relationships are external.** An entity may hold a generic attribute whose
value references an external id (e.g. an ADR id). The store neither knows nor
checks this — it only *exposes* the attribute so a caller can filter by it.
Cross-store work (coverage, dangling-ref detection) is the caller's join across
two independent stores. This store never references the other.

**Optional generic integrity.** A per-attribute registered value-set may be
enforced for a chosen attribute key (generic form validation, not domain
knowledge) to prevent typo-drift in cross-cutting attributes. The server still
does not know what that attribute *means*.

**Deployment profile (what makes this the TODO instance):** ephemeral,
per-session / per-project partition, flat file (JSON / SQLite-as-file), stored
out of git (env-configured directory), discardable. Tools exposed to the
orchestrator only.

## Consequences

### Positive
- Fits our model exactly; "you get precisely what you specified".
- Minimal footprint: no daemon, cache, or graph — a flat store.
- Removes the squash/delete dance: artifacts live outside the work tree.
- Deterministic `next` removes recency-bias as a *class* of error.
- Harness-agnostic (same protocol in OpenCode and Claude Code).
- The engine is fully generic, hence reusable.

### Negative / Costs
- We maintain it (breakage from MCP-spec / OpenCode API churn is ours).
- If a dependency graph or parallel agents ever materialize, risk of slowly
  re-deriving Beads (see Alternatives).

### Neutral
- Steal Beads' **invariants** (append-only log vs corruption on a torn write;
  hash-ids if multi-agent ever arrives), not its code.
- ~~Because the interface is generic, the same engine can back the durable (ADR)
  instance; whether to share one engine is left to implementation.~~
  **Resolved by ADR-0003:** the generic engine is shared as an internal monorepo
  module (`micro_entity`); this instance is the thin profile server
  `servers/todo.py` over it, deployed as its own idle stdio process.

## Alternatives Considered

### Beads (bd) — rejected (for now)
Mature, battle-tested; `opencode-beads` exists; stealth mode (`BEADS_DIR` +
`--stealth`) removes the git dependency. Rejected because:
1. **Complexity multiplies against a weak executor.** A dependency graph with
   four edge types, topological sort, claim semantics — extra decision surface
   in exactly the class (graph judgment) where the local orchestrator is
   weakest. Tool calibrated for agents that *can* reason over graphs; our
   offline orchestrator cannot.
2. **YAGNI.** A dependency graph and multi-agent are Beads' needs, not ours
   today. Building for "if" is over-engineering from the dependency side.
3. **Footprint.** Daemon, Unix socket, SQLite/Dolt cache, git-by-design (which
   must be *disabled*); fast version drift + a fork (steveyegge vs
   gastownhall/Dolt).

Deferral is cheap: the membership model extends by adding one more generic
attribute; adopt Beads later, knowing *why*.

### Markdown `todo.md` — rejected
Behavioral promise: the model "should" scan in order, but recency-slip breaks
it. Exactly what we are escaping.
