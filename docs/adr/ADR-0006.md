# ADR-0006: Per-project directory partitioning for both entity stores

## Status

Accepted (2026-07-01).

## Context

Both profile servers (`todo`, `adr`) currently write every record flat into a
single configured directory (`TODO_DIR` / `ADR_DIR`). Two gaps motivated a
revisit:

- **The todo profile was never actually partitioned.** ADR-0001 specified the
  TODO deployment profile as "ephemeral, per-session / **per-project** partition,
  flat file, out of git" — but the implementation is a single flat directory with
  no per-project separation. The accepted decision was simply unimplemented.
- **Cross-project mixing is a real hazard.** With one flat directory, records
  from every project the harness touches land in the same store. For the durable
  adr log this silently commingles unrelated decision logs.

Two facts shape the mechanism:

- **The MCP server cannot infer the client project from its own cwd when launched
  via `uv run --directory <repo>`**, because `--directory` chdirs the process into
  the `micro_entity_mcp` repo regardless of which workspace the harness is serving.
  The `servers` package is an installed wheel (`pythonpath = src`, editable venv),
  so `python -m servers.todo` resolves from any cwd — which means the launch can
  instead use `uv run --project <repo>` (no chdir), leaving the process cwd equal
  to the OpenCode workspace root. OpenCode launches a local MCP server with its cwd
  defaulting to the workspace.
- **A single running server serves one harness session, hence one workspace** —
  so a workspace-derived default segment is stable for the process lifetime, while
  an explicit override is still needed for cross-cutting records (e.g. one ADR that
  spans several projects).

## Decision

**Both stores partition per-project by directory, symmetrically, with the segment
resolved per operation and a hard failure when no segment can be determined.**

- **Segment resolution (precedence).** For any operation the target directory is
  `base / <segment>`, where the segment is the first of:
  1. an explicit `project` argument passed to the tool (slugified);
  2. else the workspace-derived default: `slug(basename(os.getcwd()))`, captured
     at server start;
  3. else — nothing resolves — the operation **raises** (no flat fallback).
- **Slugification.** A raw segment name is reduced to one safe path component:
  lowercased, restricted to `[a-z0-9._-]`, any other run collapsed to a single
  `-`, leading/trailing `-._` stripped, and `..` neutralized so a segment can
  never traverse outside `base`. An empty result is treated as "no segment" and
  falls through to the next precedence rule.
- **Symmetry.** `todo` and `adr` behave identically. Every data tool on both
  servers takes an optional `project: str | None = None`; `health` does not.
  A cross-cutting ADR is written by passing an explicit shared segment
  (e.g. `project="_shared"`) rather than relying on a centralized flat store.
- **Layering (partitioning is core mechanics now).** Because partitioning is now
  used *identically* by both profiles, it is shared mechanics, not divergent
  profile policy: `sanitize_segment` / `resolve_segment` and a `StoreProvider`
  live in the neutral core (`micro_entity`), and `MarkdownStore` gains an optional
  keyword-only `segment`. The core learns a generic "segment" (a directory
  component), never the domain word "project"; the servers translate their
  `project` argument into that segment.
- **Provider owns the hard-fail; the store stays permissive.**
  `MarkdownStore(base, *, segment=None)` with `segment=None` still writes flat to
  `base` (used by tests and direct callers). The refusal to run without a segment
  lives in `StoreProvider.get(project)`, surfaced to the agent as a tool error.

## Consequences

### Positive
- Implements ADR-0001's long-standing per-project TODO profile.
- Eliminates silent cross-project commingling; each project's records are isolated
  by directory.
- One generic mechanism serves both profiles (DRY), consistent with ADR-0003's
  "what both use identically lives in core."
- Cross-cutting records remain expressible via an explicit shared segment, so the
  durable log's cross-project use case (ADR-0002) is preserved as an opt-in.

### Negative / Costs
- Existing flat data under `TODO_DIR` / `ADR_DIR` (e.g. the `common_docs` dirs)
  no longer matches the new layout and must be moved under a `<segment>` subdir.
- The launch must use `uv run --project` (not `--directory`) so the process cwd is
  the workspace; a harness that forces a different cwd must pass an explicit
  `project` or it will hard-fail.
- Hard-fail (no flat fallback) means a pathological cwd with no override stops the
  operation rather than writing somewhere ambiguous — chosen deliberately over
  silent mixing.

### Neutral
- Refines ADR-0001 (implements its per-project profile and pins the mechanism),
  ADR-0002 (its "centralized, partition-by-attribute, not per-repo" stance for the
  durable store is replaced by symmetric per-project directories with an opt-in
  shared segment), and ADR-0003 (partitioning is reclassified from divergent
  profile policy to symmetric core mechanics). All three remain otherwise in force.

## Alternatives Considered

### Keep adr centralized/flat; partition only todo — rejected
Preserves ADR-0002 unchanged but leaves the two profiles asymmetric and keeps the
cross-project commingling hazard for the durable log. Rejected in favour of one
uniform mechanism with an explicit shared-segment escape hatch.

### Per-project full `*_DIR` env, no derived segment (one dir per project) — rejected
Requires hardcoding a distinct absolute path per project in per-project config and
gives no single shared root. The derived-segment approach needs only one base dir
plus an auto-derived (or overridden) subdir.

### Flat fallback when no segment resolves — rejected
Backward compatible, but reintroduces exactly the silent cross-project mixing this
record exists to remove. A hard failure is preferred: ambiguity is surfaced, not
absorbed.

### Per-tool `project` on a subset only — rejected
Asymmetric surface, harder to reason about. A uniform optional `project` on every
data tool is predictable; `health` stays param-less.
