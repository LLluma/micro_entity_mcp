# ADR-0003: Shared generic entity-store substrate and two-process topology

## Status

Accepted (2026-06-30).

## Context

ADR-0001 (ephemeral / TODO instance) and ADR-0002 (durable / ADR instance) each
decided to build a custom store, and each noted — without deciding — that the
two could share one generic engine ("left to implementation"). This record
resolves that open question and the deployment topology around it.

Observations that settle it:

- **Shared mechanics, divergent policy.** Both instances share the *access
  mechanics*: entity model (id + structural fields + bag of generic
  attributes), membership query, form validation, optional per-attribute
  registered value-set. They differ only in *policy*: lifecycle (ephemeral vs
  append-only), ordering (`next` vs none), storage (flat per-project file vs
  centralized), search (none vs full-text → vector), partitioning (per-project
  vs cross-cutting).
- **Packaging cost is real; code sharing is not.** A separately distributed
  library would impose versioning, hosting, and dual-consumer bumps — overhead
  larger than it saves for two co-located, co-deployed consumers. Sharing
  *code* (an internal module) carries none of that.
- **They run as a pair.** Both are deployed and launched together; in effect
  two instances of one thing with different configuration. Separate repos would
  invent a boundary the runtime does not have.
- **Idle cost is ~zero.** Both are stdio FastMCP servers; an idle FastMCP
  process costs effectively nothing. Ephemerality and durability live in the
  *storage* each writes (`*_DIR`), not in the process model — so there is no
  demon-vs-spawn trade to make.

## Decision

**One generic core, monorepo, two thin profile servers, two idle processes.**

- **Monorepo, internal module — not a distributed package.** Core
  `micro_entity` (entity, generic attributes, membership filter, form
  validation) lives as an internal module; `servers/todo.py` and
  `servers/adr.py` import it directly. DRY without packaging tax. One project,
  one lockfile, one environment.
- **Core stays dumb (discipline guard).** Only what both instances use
  *identically* lives in `micro_entity`. `next`/ordering → todo server.
  `supersede` chains → adr server. Cross-store joins → nowhere in any server.
  This keeps the fork-out path cheap: if the TODO instance ever grows a
  dependency graph (the one real "if" deferred in ADR-0001), fork
  `servers/todo.py`; the core is never polluted.
- **Two stdio FastMCP servers, both idle-resident.** Spawned by the harness,
  each with its own `*_DIR`. In OpenCode: two MCP blocks (`todo`, `adr`). "Let
  both run" — idle is free.
- **Topology gives structural non-intersection for free.** Two processes make a
  cross-store join *physically impossible inside a server* — no process holds
  both stores, so any `coverage` / `dangling` verdict has nowhere to live and
  must be composed by the caller. The cheap option (two free processes)
  coincides with the correct one (structural enforcement of non-intersection,
  consistent with ADR-0001/0002).
- **Code stays composable.** Written as two servers + core, so a future single
  mounted process (FastMCP `mount` / `import_server`) is a late, reversible
  config choice — not baked in.
- **Naming.** Package/core name is neutral (e.g. `micro_entity_mcp`); the
  agent-facing servers register as `todo` and `adr`. Domain words live at the
  instance level, never in the core.

## Consequences

### Positive
- DRY for the access mechanics without any packaging/hosting/versioning tax.
- Non-intersection enforced structurally at runtime, at no extra cost.
- Two cheap idle processes; trivial pair deployment from one artifact.
- Topology is reversible (mountable into one process later) thanks to
  composable code.
- Clean naming split: neutral core, domain-named instances.
- Fork-out path for a future TODO graph stays cheap (thin server, untouched
  core).

### Negative / Costs
- The monorepo removes the package boundary that would have resisted
  core-creep, so keeping the core dumb is now a *discipline*, not a structural
  guarantee. The guard rule (below) must be held by review, not by the build
  system.
- Two MCP registrations to manage per harness instead of one.

### Neutral
- Resolves the "shared engine, left to implementation" note deferred in
  ADR-0001 and ADR-0002; refines both rather than superseding them.
- The `refines` / `relates_to` attributes on this record are themselves generic
  external-link attributes (per ADR-0002) — the store does not interpret them.

## Alternatives Considered

### Separately distributed library/package — rejected
Versioning, hosting, and dual-consumer release churn for two co-located
consumers: packaging cost without an audience. "Simple as a door" core does not
justify a distributed artifact.

### Two fully independent implementations (copy / WET) — rejected
Viable if the shared core proves trivially small and stable (copying ~80 stable
lines can beat any abstraction). Rejected because the two co-deploy as a pair
and single-source dedup is wanted — but retained as the fallback if, in
implementation, the shared core turns out smaller than the wiring to share it.

### One mounted multiplexed process (single FastMCP, both toolsets) — rejected as default
Fewer moving parts, but loses the structural non-intersection that two
processes give for free, and two idle processes cost nothing anyway. Kept as a
reversible option via composable code, not chosen as the default.
