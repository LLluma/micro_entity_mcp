# ADR-0004: Core structural fields are id + timestamps only; status is a designated attribute

## Status

Accepted (2026-06-30).

## Context

ADR-0001 and ADR-0002 each described an entity as "an id + a small set of
server-understood **structural fields** (`status`, ordering/sequence, timestamps)
+ a bag of generic attributes." ADR-0003 already began narrowing that set: it
demoted **ordering** from a core structural field to a todo-server concern
("`next`/ordering → todo server"), keeping the shared core dumb. This record
finishes the same move for **status**.

Observations that settle it:

- **The core's own principle argues against a structural `status`.** ADR-0001/0002
  state the server "validates form only and **assigns no meaning** to attribute
  keys or values." A structural `status` field is the core knowing a specific
  meaning; a `status` *attribute* is the core knowing nothing — strictly more
  faithful to the stated principle.
- **Two features already in the design subsume status.** The membership `query`
  (ADR-0001) and the optional **per-attribute registered value-set** (ADR-0001,
  ADR-0002) together do everything a status enum needs: an attribute whose values
  are constrained to a registered set, and which `query` can filter by. A bespoke
  status field plus a bespoke status-enum validator would duplicate machinery the
  generic value-set already provides.
- **It removes a boundary from `query`.** With status structural, `query` ("a
  membership filter over arbitrary attributes") must either special-case status or
  refuse to filter by it. With status as an attribute, `query({status: [...]})`
  just works — no special case, no structural/attribute seam to blur.
- **Continuity with ADR-0003.** ADR-0003 demoted ordering for exactly these
  reasons; demoting status is the identical refinement, one field further, and
  keeps the "domain words live at the instance level, never in the core" rule
  whole.

## Decision

**The core's structural fields are `id` plus store-managed timestamps
(`created`, `updated`) only. `status` is a designated attribute, governed by the
generic per-attribute registered value-set and interpreted by the profile
servers. `body` is a first-class prose field. Everything else lives in the
opaque, queryable attribute bag.**

- **Structural (core/store-managed):** `id` (identity, and the record filename)
  and `created` / `updated` (timezone-aware timestamps the store sets on write).
  The core enforces their form; it assigns them no domain meaning beyond identity
  and write-time.
- **First-class prose field:** `body` (optional free text) — the full-text
  `search` target, deliberately NOT part of the membership-queryable attribute bag
  (per the body/query split adopted alongside this record).
- **Attribute bag (opaque, generic, membership-queryable):** every remaining
  key/value, **including `status`**. The core validates only form (scalar or list
  of scalars) and, where a registered value-set is configured for a key, set
  membership. It never knows that the key named "status" *means* status.
- **The enum is one instance of the registered value-set.** Each profile registers
  a value-set for its status key: the todo profile `status -> {open, done, ...}`,
  the adr profile `status -> {Proposed, Accepted, Superseded}`. The same mechanism
  guards typo-drift on any other cross-cutting attribute. There is no separate
  status-enum feature.
- **Presence is enforced at the profile layer.** Because an attribute may be
  absent, the registered value-set config carries a `required` flag; a profile
  marks its status key required and enforces "present and in-set" on create,
  update, and **load** (records are hand-editable, so load-time validation catches
  a hand-introduced typo or omission).
- **Profiles interpret status, the core does not.** `next` / `is_complete` (todo)
  and status transitions / supersede (adr) read the designated status attribute
  via profile configuration (which key holds status, which value means "done").

## Consequences

### Positive

- One mechanism, not two: the status enum is the generic registered value-set, so
  no bespoke status validator and no special-cased status field.
- `query` filters status for free; the structural/attribute boundary in `query`
  disappears.
- The shared core is dumber still — its only named fields are identity and
  timestamps; every domain-shaped value sits in the bag it assigns no meaning to.
- The build plan shrinks: the status-set validator and the registered-value-set
  feature collapse into a single generic validator.

### Negative / Costs

- Status presence is no longer guaranteed by the core model; it is enforced by the
  profile via the value-set `required` flag. The guarantee moves layers rather than
  disappearing, but it is now profile discipline + load-time validation, not a
  structural invariant.
- This deviates from the literal "status is structural" wording of ADR-0001 and
  ADR-0002. That is why it is recorded here as a refinement (those records remain
  in force, refined not superseded), so the decision log stays honest.

### Neutral

- Timestamps (`created`, `updated`) remain structural store-managed metadata, NOT
  attributes: membership-querying a timestamp is meaningless, and range queries (if
  ever wanted) are a separate feature, not the membership filter. Pushing
  timestamps into the bag was considered and declined for that reason.
- `body` remains a first-class field (the body/query split), unaffected by this
  record beyond being named here for completeness.
- The `refines` relationship on this record is a generic external-link attribute
  (per ADR-0002); the store does not interpret it.

## Alternatives Considered

### Keep `status` as a core structural field (ADR-0001/0002 literal) — rejected
Requires a bespoke status-enum validator separate from the registered value-set,
and forces `query` to either special-case status or refuse to filter by it. Two
mechanisms where one suffices, and a seam in the otherwise-uniform attribute model.

### Demote timestamps to attributes as well — rejected
Timestamps are store-managed metadata and are not usefully membership-queryable;
folding them into the bag would invite meaningless `query({created: [...]})` calls
and blur the store's ownership of write-time. Kept structural.
