# ADR-0005: Durable adr store record schema and status lifecycle

## Status

Accepted (2026-06-30).

## Context

The durable (decision-log) profile of ADR-0002 must dogfood the project's own
decision records: "this very record will be among the first the store holds." That
makes the on-disk shape of these very files (`docs/adr/ADR-NNNN.md`) the de facto
schema the adr profile has to read and validate. Three different "ADR shapes"
existed in the project and disagreed, so the schema had to be pinned:

- **On disk (ADR-0001…0004, the actual files).** Rich YAML frontmatter (`title`,
  `tags`, `supersedes`, `superseded_by`, `refines`, `refined_by`, `relates_to`,
  `date`) plus a prose body (`## Context / Decision / Consequences / Alternatives`).
  Status values in use: `Proposed` → `Accepted`.
- **ADR-0002's own description.** Generic attributes (project, tags, supersedes
  pointer, relates_to) with a status enum of `Proposed / Accepted / Superseded`.
- **An orchestration convention (`adr-format.md`).** A minimal, frontmatter-less
  shape (`# ADR-NNNN — DATE — title`, a `Status:` line, `Decision / Rationale /
  Rejected`) with statuses `Active / Planned / Superseded`.

Two observations settle the choice:

- **The entity model requires the rich frontmatter.** The adr store exists to
  expose a queryable attribute bag; `query` operates on attributes. The minimal
  convention has no frontmatter, so adopting it would leave the store with almost
  nothing to query — gutting the very purpose of the durable store. The rich
  frontmatter, by contrast, maps cleanly onto the core entity (ADR-0004): `id` =
  filename, timestamps structural, `body` = prose, and `tags` / `supersedes` /
  `superseded_by` / `refines` / `refined_by` / `relates_to` / `title` = the
  queryable attribute bag.
- **The minimal convention is being retired.** The orchestration `adr-format.md`
  contract is an outdated solution slated for removal; the durable adr store
  replaces it. With it gone there is no competing schema — the on-disk rich shape
  is the single source of truth.

## Decision

**The rich-frontmatter on-disk shape is the canonical record schema for the durable
adr profile, with status enum `{Proposed, Accepted, Superseded}` and supersession
expressed as an attribute pointer.**

- **Schema = the on-disk rich shape, mapped per ADR-0004.** Per record:
  - `id` — the filename stem (`ADR-0007`); not duplicated in frontmatter.
  - `created` / `updated` — structural tz-aware timestamps (see migration below).
  - `body` — the full prose (`## Context / Decision / Consequences / Alternatives`),
    the `search` target, not a queryable attribute.
  - attribute bag — every other frontmatter key: `title`, `tags`, `supersedes`,
    `superseded_by`, `refines`, `refined_by`, `relates_to`, and any future
    cross-cutting key. The store assigns them no meaning; an external reader
    interprets the link attributes (ADR-0002).
- **Status is a designated attribute (ADR-0004), enum `{Proposed, Accepted,
  Superseded}`**, registered as a required value-set on the adr profile. The core
  knows none of these values.
- **Supersession is an attribute pointer, not a status string.** A superseded
  record has `status: Superseded` plus a `superseded_by: ADR-NNNN` attribute; the
  superseding record carries `supersedes: ADR-NNNN`. The target id lives in a
  queryable attribute, NOT baked into the status value (rejecting the
  `Superseded by ADR-NNNN` form). Status stays a clean enum; the link stays a
  separate attribute.
- **Append-only discipline (ADR-0002, profile discipline per the store decision).**
  The adr server exposes no delete tool; a changed decision is a new record plus a
  `Superseded` status on the old one. Status transitions and hand-edits are allowed
  (lossless round-trip).
- **Timestamp migration.** Existing files carry `date:` but not `created`/`updated`.
  On load, when `created`/`updated` are absent, derive both from `date` (interpreted
  as midnight UTC); the store writes explicit `created`/`updated` from then on. A
  record that violates the schema on load is quarantined and reported, never fatal
  (the quarantine load model).
- **The product adr store is a distinct concern from any orchestration ADR
  convention.** Its `ADR_DIR` is configurable; dogfooding points it at `docs/adr/`,
  which conforms because those files already use the rich shape. The retired
  `adr-format.md` convention is not a consumer of this store.

## Consequences

### Positive

- The store dogfoods the existing decision log with no import or rewrite step — the
  files already are the schema.
- A wide, queryable attribute surface (`tags`, `relates_to`, `refines`,
  `supersedes`, …) — the durable store's reason to exist — is preserved.
- Status stays a clean three-value enum; supersession is a first-class queryable
  link rather than a parsed status string.
- One less convention to maintain: the retired minimal format is replaced, not
  duplicated.

### Negative / Costs

- A small load-time migration (derive `created`/`updated` from `date`) is carried
  indefinitely for legacy records, or until they are rewritten with explicit
  timestamps.
- The rich schema is heavier to author by hand than the minimal convention was —
  acceptable, since records are few and durable.

### Neutral

- Refines ADR-0002 (which left the concrete schema and enum open); ADR-0002 remains
  in force.
- The `refines` / `relates_to` attributes on this record are themselves generic
  external-link attributes (ADR-0002, ADR-0004) — the store does not interpret them.
- The concrete prose section set (`Context / Decision / Consequences /
  Alternatives`) is a human authoring convention for the body; the store treats the
  body as opaque text and does not validate its internal structure.

## Alternatives Considered

### Adopt the minimal `adr-format.md` convention as the schema — rejected
Frontmatter-less, so the store would have almost no attributes to `query` — it
removes the durable store's whole purpose. Also the convention is being retired, so
aligning to it would target a format on its way out.

### Bake the supersession target into the status (`Superseded by ADR-NNNN`) — rejected
Turns a clean enum value into a parsed string and hides a cross-record link inside
status. Expressing it as a `superseded_by` attribute keeps status a plain
value-set member and the link a first-class queryable attribute.

### Migrate all existing ADR files to a new canonical format — rejected
The on-disk rich shape already maps onto the entity model; rewriting the decision
log to satisfy the store would invert the dogfooding relationship (the store should
read the records as they are, not force a churn of the log).
