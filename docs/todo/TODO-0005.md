# TODO-0005 — todo profile server (ephemeral)
ADR: ADR-0001, ADR-0004
Status: active — 10 items (2 done, 8 todo)

Settled decisions for this batch (items G, H-todo, J, K):
- Thin FastMCP stdio server `src/servers/todo.py` over the shared core + MarkdownStore,
  its own `TODO_DIR` (env-configured), per-session partition, discardable. Core stays
  dumb; all todo policy lives here (ADR-0003).
- IDs: SERVER-GENERATED SEQUENTIAL — scan partition, assign next zero-padded int.
- ORDER: a separate mutable integer attribute `order`, distinct from id; `next` sorts
  by it; reprioritizing changes `order`, not id.
- STATUS enum: `{todo, in-progress, done, blocked}`, a required value-set (the generic
  validator, ADR-0004), enforced on create/update.
- `next`: deterministic — first entity with status in {todo, in-progress} by `order`
  (sorted[0]); SKIPS `done` AND `blocked`. Ignores `depends_on` (no dependency graph).
- `is_complete`: True iff no entity is in {todo, in-progress, blocked} (all `done`).
- Errors (J): FormError / not-found → structured MCP tool errors; quarantine surfaced.
- Tests (K): FastMCP in-memory `Client(server)` transport, no subprocess.

## 1. Server skeleton + config + test harness — [done]
Goal: a runnable FastMCP server bound to a MarkdownStore over `TODO_DIR`, with the
status value-set constant defined, and an in-memory client fixture for the tool tests.
Interface: `src/servers/todo.py` exposing a FastMCP app `mcp`; `TODO_DIR` read from
env (default a sensible path); module constant `STATUS_VALUES = {"todo","in-progress",
"done","blocked"}` and `STATUS_KEY = "status"`, `ORDER_KEY = "order"`. A pytest fixture
spins the app with a `tmp_path` store via the in-memory client.
Acceptance: the app starts; an empty `list`/ping-style tool returns over the in-memory
client; STATUS_VALUES present. No record tools yet beyond a trivial health/list stub.
Constraints: store directory from `TODO_DIR`; reuse MarkdownStore.
File scope: create `src/servers/todo.py` and `tests/test_todo_server.py` only.

## 2. Sequential id generator — [done]
Goal: derive the next id by scanning existing record ids in the partition.
Interface: a helper `def _next_id(store) -> str` — find the max integer stem among
existing `*.md`, return the next as a zero-padded string (e.g. `0001`); first id on an
empty partition is `0001`.
Acceptance: empty partition → `0001`; with `0001,0002` → `0003`; padding width stable;
ignores non-integer stems (or defines them out — state the rule).
Constraints: pure-ish (reads the store/partition); no global counter file.
File scope: edit `src/servers/todo.py` and its test only.

## 3. create tool — [todo]
Goal: add a new todo item with a generated id, default status, and an assigned order.
Interface: tool `create(body: str, attributes: dict | None = None) -> dict` — generate
id (item 2); set `status="todo"` unless provided (validate against STATUS_VALUES);
assign `order = max(order)+1` (or 1 if empty); persist via store.create; return the
entity as a dict.
Acceptance: create returns an entity with a fresh id, status `todo`, next `order`,
the given body; an invalid status in attributes → structured error; order increments
across creates.
Constraints: status enforced via `validate_against_set`; order auto-assigned.
File scope: edit `src/servers/todo.py` and its test only.
