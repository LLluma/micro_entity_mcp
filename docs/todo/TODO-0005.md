# TODO-0005 — todo profile server (ephemeral)
ADR: ADR-0001, ADR-0004
Status: active — items TBD (specs written when batch is reached)

Settled decisions for this batch (items G, H-todo, J, K):
- Thin FastMCP stdio server `servers/todo.py` over the shared core + MarkdownStore.
  Its own `TODO_DIR` (env-configured), per-session/per-project partition,
  discardable. Core stays dumb; all todo policy lives here (ADR-0003).
- IDs: SERVER-GENERATED SEQUENTIAL (item G-1). Server scans partition, assigns next
  integer id (zero-padded). Agent never invents ids.
- ORDER (item G-2): a separate mutable integer attribute `order`, distinct from id
  (stable identity). `next` sorts by `order`. Reprioritizing changes `order`, not id.
- STATUS enum (item H-todo): `{todo, in-progress, done, blocked}`, registered as a
  required value-set (the generic validator, ADR-0004). Enforced on create/update/
  load.
- `next` (item G-3, H-todo): deterministic — first entity whose status ∈
  {todo, in-progress} by `order` (sorted[0]); SKIPS `done` AND `blocked`. Never
  "judges the list" (ADR-0001). Ignores `depends_on` entirely (no dependency graph).
- `is_complete`: True iff no entity is in {todo, in-progress, blocked} (i.e. all
  `done`). A `blocked` item means NOT complete and NOT pickable.
- `depends_on` (if used) is exposed generic data only; never resolved by the server.
- Tools (one atomic item each): server skeleton; create; get; list; update(status);
  delete; clear; next; is_complete; query (delegates to TODO-0004).
- Errors (item J): FormError / not-found → structured MCP tool errors (never
  swallowed); quarantine `errors` surfaced, not fatal.
- Tests (item K): FastMCP in-memory `Client(server)` transport, no subprocess.
