# TODO-0003 — micro_entity: MarkdownStore (directory partition)
ADR: ADR-0003
Status: active — items TBD (specs written when batch is reached)

Settled decisions for this batch (from session clarification):
- One concrete `MarkdownStore` serves BOTH profiles; partition = a directory, one
  file per entity named `{id}.md`. `id` = filename stem (round-trips to the Entity).
- Store-managed timestamps via an INJECTABLE CLOCK (item F): the store takes
  `clock: Callable[[], datetime] = lambda: datetime.now(UTC)`. `create` sets
  `created` + `updated`; `update` bumps `updated` only.
- QUARANTINE load model (item E): `load`/`list` returns `(valid_entities, errors)`,
  NOT a bare list. A record that fails form/value-set validation on load is skipped
  and reported in `errors` (id + reason), never fatal. Records are hand-editable, so
  load-time validation catches hand-introduced typos.
- Atomic writes: temp-file + rename (ADR-0001 stolen-Beads invariant: no corruption
  on a torn write). No locking (item L: single-user, per-DIR).
- Store is CAPABLE of delete/clear (item C). Append-only is NOT enforced here — it
  is the adr profile's tool-surface discipline (TODO-0006).
- The store is the seam where a registered value-set may be enforced on load/write
  (the generic validator from TODO-0001 item 4); WHICH key/set is profile config,
  passed in — the store/core bakes in no concrete attribute names or values.
- Storage backend is a `Store` Protocol in core + this one `MarkdownStore` impl, so
  a future SQLite/vector backend is additive (ADR-0003 composability).

Item decomposition (atomic, full specs when reached):
- Store Protocol definition in core.
- MarkdownStore: create (server-managed timestamps via clock).
- MarkdownStore: get (not-found contract).
- MarkdownStore: list/load → (valid, errors) quarantine.
- MarkdownStore: update (bump updated; status/attr edits).
- MarkdownStore: delete / clear.
- Atomic temp+rename write helper.
