## 1. adr write path breaks on legacy (date-only) records — [done]
Goal: make `update` and `supersede` work against records that carry only a legacy
`date:` (no `created`/`updated`) — today they raise `CodecError`, so the whole
supersession lifecycle is unusable against the real `docs/adr/` log the profile is
built to dogfood.
Evidence (verified): with a copy of `docs/adr`,
`MarkdownStore.update("ADR-0001", attributes={"status":"Superseded"})` →
`CodecError: missing required timestamp: created`. Reason: migration is wired only to
the READ path (`_get_migrated` / `_load_all_migrated` in `src/servers/adr.py`), while
`store.update` reads through the core codec (`parse_document` → `entity_from_parts`),
which mandates `created`/`updated`. `supersede` (adr.py:241-248) calls `store.update`,
so it fails identically. `get` also never persists the derived timestamps, so
ADR-0005's promise "the store writes explicit created/updated from then on" is unmet.
Interface: the adr `update` and `supersede` tools must apply the same
`_normalize_frontmatter` migration on the write path before the core codec sees the
frontmatter (e.g. a migrating update helper in adr.py, or a normalize hook the store
update accepts). No new tool signatures.
Acceptance: on a tmp copy of `docs/adr`, `update("ADR-0001", status="Superseded")`
succeeds, writes explicit `created`/`updated` derived from `date` (midnight UTC),
drops/keeps `date` consistently with the read path, and preserves comment/key order;
`supersede("ADR-0001","ADR-0002")` succeeds end-to-end on legacy records; a record
already carrying explicit timestamps is unchanged by the migration.
Governing decisions (ADR-0005, inline): legacy files carry `date:` but not
created/updated; on load derive both from `date` as midnight UTC and the store writes
explicit created/updated from then on; a schema-violating record is quarantined, never
fatal. Supersession is an attribute pointer (`superseded_by` / `supersedes`), never a
baked status string; the profile is append-only (no delete).
Constraints: migration stays adr-PROFILE policy — do NOT push `date` handling into the
core codec/entity. Read-only against the real `docs/adr/` in tests (use a tmp copy for
any write path).
Implementation sketch: add a private `_update_migrated(store, id, *, attributes=None,
body=UNSET) -> Entity` in adr.py that mirrors `MarkdownStore.update` but runs the
frontmatter through `_normalize_frontmatter` + `fm.pop("date", None)` after
`parse_document` and before `serialize_document`/`entity_from_parts` — i.e. the same
normalize step `_get_migrated` already applies on read, now on the write path. Route
the `update` tool and BOTH `store.update` calls in `supersede` through it. (Do not
special-case in core `MarkdownStore.update`; keep `date` knowledge in the profile. This
helper is superseded by item 5's normalize-hook refactor — leave a note so item 5 folds
it in.) Tests: build them from a tmp copy of `docs/adr` (`shutil.copytree(tmp_path)`),
never the real dir.
Done means: tests green AND ruff/pyright clean.
How to run: `make test`, `make lint` (see Makefile).
File scope: `src/servers/adr.py`, `tests/test_adr_server.py` only.

## 2. Store Protocol contract diverges from MarkdownStore; test_store.py over-mocks — [done]
Goal: make the declared `Store` seam (`src/micro_entity/store.py`) and the concrete
`MarkdownStore` agree, so code written against the Protocol is actually correct — and
remove the tautological `_StubStore` tests that assert against a fake instead of real
code.
Evidence (contract): Protocol docstrings say `create` raises `ValueError` on exists,
`get` / `delete` raise `ValueError` when not found — MarkdownStore raises
`FileExistsError` / `NotFoundError`. Protocol `update` signature is `body: str | None =
None`; MarkdownStore is `body: str | None | UnsetType = UNSET`. So MarkdownStore does NOT
structurally satisfy the declared type (the `body` parameter type differs), and a caller
trusting the Protocol's `ValueError` contract would miss `NotFoundError`.
`@runtime_checkable` hides this because it only checks method NAMES, not signatures or
error contracts.
Evidence (over-mock): `_StubStore` (test_store.py:50) is a do-nothing fake whose
`create`/`update` echo their arguments back into an `Entity`. Six tests assert on that
echo and exercise ZERO production code: `test_create_signature_accepts_scalar_types`
(:139), `test_create_default_body_none` (:149), `test_update_preserves_existing` (:164 —
worst: the stub holds no state, so it cannot possibly "preserve omitted fields" as its
name claims), `test_load_all_returns_typed_tuple` (:155), `test_delete_is_none_return`
(:171), `test_clear_is_none_return` (:177). Every one of these behaviors is already
tested against the REAL store in test_markdown_store.py (create :12,44; body/default
:258,272,282; update-preserve :301; delete :458,474; clear :487,506,513; load_all
:383,399), so the stub versions are redundant as well as vacuous.
Interface: pick one contract and align both. Recommended: Protocol declares the real
error types (`NotFoundError`, `FileExistsError`) and the real `update` body sentinel
(expose `UNSET`/`UnsetType` from a shared spot, or fold the sentinel into the Protocol).
Acceptance: the Protocol's documented errors and `update` signature match MarkdownStore
exactly; a `MarkdownStore` instance still passes `isinstance(..., Store)`; pyright clean.
The 6 tautological stub tests are gone; the structural-conformance test now uses the REAL
`MarkdownStore` as its `isinstance` witness; the remaining Protocol-shape tests
(runtime-checkable, method-names, docstrings-present, not-abc, and the one stub test that
proves a non-subclass satisfies the Protocol structurally) still pass.
Constraints: do not weaken the store's actual behavior to match a wrong doc — fix the
contract to the behavior. Don't change tool-facing behavior. Do NOT delete the genuine
Protocol-contract tests, and keep exactly one `_StubStore`-based test to prove structural
(non-subclass) conformance — that is the stub's one legitimate job.
Implementation sketch: (a) move `UnsetType`/`UNSET` out of `markdown_store.py` into a
neutral spot both files import — either `store.py` or a new `_sentinel` — to avoid a
`store.py → markdown_store.py` import cycle. (b) In the `Store` Protocol change the
`update` body param to `str | None | UnsetType = UNSET` and import the sentinel. (c) Fix
the docstrings: `create` "Raises `FileExistsError` if *id* already exists"; `get` /
`update` / `delete` "Raises `NotFoundError` if *id* does not exist" (import/reference
`NotFoundError`, already in `store.py`). (d) Add a `test_store.py` assertion that
`isinstance(MarkdownStore(tmp_path), Store)` holds (real implementation is the witness),
and that a Protocol-typed variable accepts a MarkdownStore under pyright. (e) DELETE the
6 tautological stub tests listed in the evidence; keep `_StubStore` only for the single
non-subclass structural-conformance test, and keep all the Store-introspection tests
(names/docstrings/not-abc). Confirm coverage is unchanged (the deleted behaviors remain
covered by test_markdown_store.py).
Done means: tests green AND ruff/pyright clean.
How to run: `make test`, `make lint`.
File scope: `src/micro_entity/store.py`, `src/micro_entity/markdown_store.py` (sentinel
move only), `tests/test_store.py`; adjust any test asserting the old ValueError contract.

## 3. Reserved frontmatter keys are clobberable via the attributes bag — [done]
Goal: stop callers from overwriting structural/reserved frontmatter through the generic
`attributes` dict on the adr/todo tools.
Evidence: both servers pass user `attributes` straight into `store.create`/`store.update`,
which writes each key into frontmatter (`markdown_store.py:173-174` `fm[k]=v`). A caller
can set `attributes={"created": "..."}` (clobbers the structural timestamp — `updated` is
re-stamped by the clock but `created` is not protected) or inject other reserved keys.
In adr `add`, a raw `attributes={"status": ...}` is at least routed through the enum
check, but `created`/`updated` are not guarded anywhere.
Interface: reject (or strip) reserved keys `created`, `updated` (and `id`) from the
user-supplied `attributes` in the adr and todo tool handlers, raising `ToolError` on an
attempt to set them.
Acceptance: `add`/`create`/`update` with `attributes={"created": ...}` (or `updated`,
`id`) → `ToolError`; ordinary attributes still pass through; existing tests green.
Constraints: enforce at the profile/tool layer; keep the core store dumb.
Implementation sketch: define `RESERVED_KEYS = {"created", "updated", "id"}` in each
server. In adr `add`/`update` and todo `create`/`update`, after building the `attrs`/
`patch` dict, do `bad = RESERVED_KEYS & attrs.keys(); if bad: raise ToolError(f"cannot
set reserved keys: {sorted(bad)}")` BEFORE the status/order handling injects its own
keys. Note `status` and `order` arrive as their own typed params and are legitimately
written — only the free-form `attributes` bag is screened. Add one negative test per
reserved key per tool.
Done means: tests green AND ruff/pyright clean.
How to run: `make test`, `make lint`.
File scope: `src/servers/adr.py`, `src/servers/todo.py`, and their tests only.

## 4. supersede is non-atomic — partial failure leaves a half-linked pair — [done]
Goal: ensure `supersede(old_id, new_id)` cannot leave the append-only log in a
half-updated state.
Evidence: adr.py:241-248 does two independent `store.update` calls. If the second
(`supersedes` on the new record) fails, the old record is already `Superseded` +
`superseded_by=new_id` while the new record has no back-pointer — an inconsistent,
un-rolled-back state in a durable decision log.
Interface: make the pair all-or-nothing — validate both records fully up front (already
partly done via `_path_for` existence checks), then apply both writes with rollback of
the first if the second raises, or otherwise guarantee both-or-neither.
Acceptance: when the second update is forced to fail (injected), the old record is
restored to its pre-call state (no dangling `Superseded`); the happy path is unchanged;
both-missing / one-missing still error before any write.
Constraints: no new store transaction machinery required — a compensating restore of the
first record is acceptable given single-writer stdio use.
Implementation sketch: capture the old record's pre-call frontmatter first (read the raw
`.md` text of `old_id`, or snapshot the Entity), apply the `old_id` write, then apply the
`new_id` write inside `try`; on exception in the second write, re-write the captured
old-record text back and re-raise as `ToolError`. Test with monkeypatch: patch
`store.update` (or the file write) to raise on the `new_id` call and assert `old_id`'s
on-disk `status` is unchanged and no `superseded_by` was left behind. Depends on item 1
(supersede must first work on legacy records at all).
Done means: tests green AND ruff/pyright clean.
How to run: `make test`, `make lint`.
File scope: `src/servers/adr.py`, `tests/test_adr_server.py` only.

## 7. adr `get` crashes (unhandled) on a malformed legacy `date` — [done]
Goal: a single record with a mangled `date:` frontmatter value must not take down the
`get` tool; it should surface as a clean tool error (and quarantine like the list path).
Evidence (verified): a record `date: not-a-real-date` →
`_normalize_frontmatter` calls `date_cls.fromisoformat(raw_date[:10])` (adr.py:56)
unguarded → bare `ValueError: Invalid isoformat string`. The `get` tool (adr.py:188-193)
only catches `NotFoundError` and `FormError`, so this `ValueError` (and any `CodecError`
from the codec) propagates unhandled. `list`/`query`/`search` survive because
`_load_all_migrated` wraps each record in `except Exception` — `get` is the inconsistent
gap.
Interface: `get` must convert record-parse failures into `ToolError`; the migration
helper should fail gracefully on an unparseable `date`.
Acceptance: `get` on a record with a malformed `date` → `ToolError` (not a raw
exception / 500); `get` on a well-formed legacy record still migrates and returns; the
`list`/`query`/`search` quarantine behavior is unchanged.
Constraints: keep migration adr-profile policy; do not swallow the reason — the
`ToolError` message must carry the underlying cause.
Implementation sketch: in `_normalize_frontmatter`, wrap the `str` branch's
`date_cls.fromisoformat(...)` in `try/except ValueError: return fm` (leave un-normalized
so the downstream codec quarantines/reports it uniformly); AND widen the `get` tool's
except to `except (NotFoundError, FormError, CodecError, ValueError) as e: raise
ToolError(str(e)) from e`. Add a test with a malformed-`date` fixture asserting `get`
raises `ToolError`.
Done means: tests green AND ruff/pyright clean.
How to run: `make test`, `make lint`.
